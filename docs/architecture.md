# Архитектура

## Границы ответственности

- **Bedolaga** остаётся источником истины для оплат, тарифов и оплаченных
  подписок.
- **Grace Bridge** владеет только временным техническим overlay после окончания
  оплаченной подписки.
- **Remnawave** исполняет рассчитанное состояние пользователя.
- **Grace-нода** ограничивает маршруты и скорость. Bridge не подменяет сетевую
  политику.

## Состояние и очередь

Bridge создаёт отдельную схему PostgreSQL `grace_bridge`. Она не включается в
Alembic Bedolaga и поэтому не конфликтует с её номерами миграций.

Ключевые механизмы:

1. Advisory lock берётся по `user_id`, а не по отдельной исторической подписке.
2. Каждое изменение повышает `generation`.
3. Команда уникальна по `(subscription_id, generation, action)`.
4. Worker повторно проверяет право на Grace под тем же lock.
5. Запись в Remnawave подтверждается чтением после записи.
6. Оплата повышает `generation`, делает старые команды `stale`, после чего
   обычная логика Bedolaga получает приоритет.

## Массовое включение

Процент выбирается детерминированно по UUID. Один пользователь остаётся в той
же когорте, а множества вложены друг в друга:

```text
5% ⊂ 25% ⊂ 50% ⊂ 100%
```

Это предотвращает случайную смену тестовой когорты после перезапуска.

## Отказобезопасность

- `disabled` — нет подключения к БД и нет записей;
- `observe` — read-only транзакции;
- `canary` — записи разрешены только для одного UUID;
- `active` — требуется подтверждение и конкретная ступень процента;
- `pause` — немедленно прекращает новые активации;
- `rollback` — сначала снимает только overlay с тегом `GRACE_ACCESS`, затем
  возвращает исходный образ Bedolaga.
