# Нагрузочное тестирование

Тест использует только отдельную PostgreSQL-базу с обязательным именем
`grace_bridge_loadtest`. Дополнительно требуется фраза
`GRACE_LOAD_TEST_CONFIRM=ISOLATED_GRACE_BRIDGE_LOADTEST`. SQL-сканер вообще не
создаёт клиента Remnawave. Конкурентный тест worker'ов направляет клиент только
на локальный фальшивый HTTP API внутри тестового контейнера; адрес и ключ
настоящей панели ему не передаются.

Для изолированного сервера предусмотрена команда:

```bash
sudo bash scripts/run_isolated_remote_loadtest.sh
```

Она создаёт временную internal Docker network, PostgreSQL на `tmpfs` и локальный
образ контроллера. Контейнеры ограничены 1 CPU и 512/384 MiB памяти. Trap удаляет
контейнер, сеть и тестовый образ как при успехе, так и при ошибке.

## Контрольный результат v0.1.0

Проверка от 19 июля 2026 года на 40 000 синтетических записей:

| Показатель | Результат |
|---|---:|
| Пользователи | 40 000 |
| Размер страницы | 500 |
| Страницы | 80 |
| Подготовка базы | 5.839 с |
| Полный keyset-скан | 17.695 с |
| Скорость сканирования | 2260.5 кандидата/с |
| Внешние API | не использовались |
| Повторные UUID | 0 |

В том же изолированном запуске проверена конкуренция четырёх worker'ов на
1 000 синтетических пользователей:

| Показатель | Результат |
|---|---:|
| Успешные активации | 1 000 |
| Скорость активации | 67.6 пользователя/с |
| Максимальная конкуренция фальшивого API | 4 |
| Зависшие команды | 0 |
| Потерянные снимки для компенсации | 0 |
| Оставшиеся advisory-lock | 0 |
| PATCH и сброс трафика | ровно по одному на UUID |

Это проверка SQL-выборки и пагинации, а не разрешение немедленно включать
40 000 пользователей. Производственный rollout всё равно ограничен ступенями
5% → 25% → 50% → 100% после отдельного canary.
